## 1. Context and Diagnosis

After the first code iteration, three categories of issues were identified that block the application from being usable.

### 1.1 Critical correction — 2026 World Cup format

RFC-001 and the first code iteration assumed **32 teams and 8 groups** (the classic format used in 1998–2022). The 2026 World Cup is the **first ever 48-team tournament**, which makes the old `wc32` template wrong in both group count and knockout structure.

**2026 World Cup format (confirmed by FIFA):**

| Element | Value |
|---------|-------|
| Teams | 48 |
| Groups | 12 groups of 4 |
| Advancement | Top 2 from each group (24) + best 8 third-placed sides = **32 teams** in knockout |
| Knockout rounds | Round of 32 → Round of 16 → Quarter-finals → Semi-finals → 3rd place play-off → Final |
| Total matches | 104 (all-time record) |
| Finalists play | 8 matches (previously 7) |

**Impact on the app:**

1. The primary template ID changes from `wc32` → **`t48`** (48-team tournament, 12 groups, 7 phases including R32)
2. The old `t32` template (8 groups, 6 phases) is kept as a secondary "classic" option
3. The country picker must offer **48 countries**, not 32
4. The "best 8 third-placed teams" mechanic is **not automated** — the owner manually creates Round of 32 matches after the group stage, picking which 8 third-placed sides advance

### 1.2 Template naming convention

All template IDs and display names must be **tournament-agnostic** — no "World Cup", "MŚ", or any competition branding. The app is a generic pool tool; the owner names their own tournament.

| Old ID | Old name | New ID | New display name (i18n value in pl.json) |
|--------|----------|--------|------------------------------------------|
| `wc48` | MŚ 2026 — 48 drużyn | `t48` | Turniej — 48 drużyn |
| `wc32` | World Cup 32 | `t32` | Turniej — 32 drużyny |
| `wc16` | World Cup 16 | `t16` | Turniej — 16 drużyn |
| `league` | League | `league` | Liga |
| `custom` | Custom | `custom` | Własny turniej |

`t48` is the **default** and shown first in the template selector.

## 2. Bug Analysis (root cause)

### BUG-001 — Persistence is broken (data lost on reload)

**File:** `src/store/tournamentStore.ts`

**Cause:** The Zustand `persist` middleware uses a custom `tauriStorage` adapter backed by `@tauri-apps/plugin-store`. In dev mode (Vite without Tauri runtime), `createStore` from `@tauri-apps/plugin-store` does not exist and throws, so the persist middleware never writes state. The `tauriStorage` object is also constructed at module load time before the async store is ready, creating a race condition even in production Tauri builds.

**Fix:** Extract storage into a dedicated `src/store/storage.ts` file that detects the environment and falls back to `localStorage` when running outside Tauri.

```typescript
// src/store/storage.ts — new file

import { StateStorage } from 'zustand/middleware';

const isTauri = typeof window !== 'undefined' && '__TAURI__' in window;

// Lazy Tauri store initialisation
let _tauriStore: any = null;
async function getTauriStore() {
  if (!_tauriStore) {
    const { createStore } = await import('@tauri-apps/plugin-store');
    _tauriStore = await createStore('libero.store');
  }
  return _tauriStore;
}

const tauriStorage: StateStorage = {
  getItem: async (name) => {
    const store = await getTauriStore();
    const data = await store.get(name);
    return data ? JSON.stringify(data) : null;
  },
  setItem: async (name, value) => {
    const store = await getTauriStore();
    await store.set(name, JSON.parse(value));
    await store.save();
  },
  removeItem: async (name) => {
    const store = await getTauriStore();
    await store.delete(name);
    await store.save();
  },
};

const localStorageAdapter: StateStorage = {
  getItem: (name) => localStorage.getItem(name),
  setItem: (name, value) => localStorage.setItem(name, value),
  removeItem: (name) => localStorage.removeItem(name),
};

export const appStorage: StateStorage = isTauri ? tauriStorage : localStorageAdapter;
```

Also verify: `tauri-plugin-store` is listed in `src-tauri/Cargo.toml` and registered in `src-tauri/src/lib.rs`.

### BUG-002 — Match pick form exists but is never rendered

**Files:** `src/App.tsx`, `src/features/picks/MatchPickForm.tsx`

**Cause:** `MatchPickForm` is implemented but **never used anywhere**. `App.tsx` only renders `TournamentPickForm` (champion + top scorer) per player. The entire flow of **player → list of matches → enter pick** is missing from the UI.

**Fix:** New `PlayerPicksSection` component, rendered in the Picks view. Each player gets an accordion that expands to show tournament picks + all matches grouped by phase, each with an inline `MatchPickRow`.

See section 4 for component details.

---

### BUG-003 — `TournamentPhase.type` is a hardcoded enum instead of dynamic flags

**Files:** `src/types/index.ts`, `src/store/tournamentStore.ts`

**Cause:** `TournamentPhase` has `type: 'group' | 'round_of_16' | 'quarter' | 'semi' | 'final'` — a hardcoded enum that contradicts RFC-001 §7, which specifies dynamic phases with `isKnockout: boolean`. The store scoring logic uses `phase.type === 'group'` to decide between group-stage and knockout point calculation, which breaks for any custom template.

**Fix:** Replace `type` with `isKnockout` + `allowDraw` + `label` + `order` as specified in RFC-001.

```typescript
// BEFORE (wrong)
interface TournamentPhase {
  type: 'group' | 'round_of_16' | 'quarter' | 'semi' | 'final';
  matches: Match[];
}

// AFTER (RFC-001 §7)
interface TournamentPhase {
  id: string;
  label: string;       // user-defined, not an i18n key
  order: number;
  isKnockout: boolean;
  allowDraw: boolean;
  matches: Match[];
}
```

Scoring dispatch in the store becomes:

```typescript
const phase = tournament.phases.find(p => p.matches.some(m => m.id === pick.matchId));
const points = phase?.isKnockout
  ? calculateKnockoutPhasePick(pick, result, homeTeam, awayTeam)
  : calculateGroupPhasePick(pick, result);
```

### BUG-004 — Tournament creator always generates hardcoded demo data

**File:** `src/features/tournament/TournamentCreator.tsx` — `handleCreate`

**Cause:** `handleCreate` ignores the selected template and always creates the same two demo phases with four hardcoded countries (PL, DE, FR, ES). The user cannot create a real tournament.

**Fix:** `handleCreate` calls `applyTemplate(templateId)` from `src/data/templates.ts` to build the correct phase structure and empty groups. `TournamentForm` gains a country picker loaded from `src/data/countries.ts` (48 countries).

### MISSING-001 — No sidebar (horizontal tab bar instead)

**File:** `src/App.tsx`

**Cause:** RFC-001 specifies a sidebar layout, but the first iteration implemented a horizontal tab bar. On a desktop Tauri window the tab bar is awkward and will run out of space as sections grow.

**Fix:** See section 3.
