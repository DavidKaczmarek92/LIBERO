# RFC-001: LIBERO — Football World Cup Betting Pool Application

**Status:** Draft  
**Author:** TBD  
**Date:** 2025  
**Stack:** Tauri 2 · React 18 · TypeScript 5 · Tailwind CSS 3 · React Flow · i18next · Vitest · Playwright

---

## 1. Context and Goal

Libero Law Firm runs an internal betting pool for the Football World Cup. The app runs locally on a single machine — no login, no roles. The laptop owner = administrator = the only person operating the app. Players trust each other to act honestly.

Goal: replace a manual spreadsheet / Messenger thread with a desktop app that automatically calculates points, renders a visual tournament bracket, and provides a convenient picks form.

---

## 2. Design Decisions (abbreviated ADR)

| # | Decision | Rationale |
|---|---|---|
| 1 | No pick locking | Players self-enforce honesty; the laptop owner can edit everything at any time |
| 2 | No result locking | Owner can correct a wrongly entered result at any time; points recalculate automatically |
| 3 | No login / roles | Single operator, simplicity over access control |
| 4 | Code + tests in English | Standard practice; easier to search for errors, consistent with library docs |
| 5 | UI exclusively in Polish via i18next | Polish as the only language for now; i18n keys in English in code |
| 6 | React Flow for visualisation | Knockout bracket, tournament creator, group stage view — one consistent graph renderer |

---

## 3. Scope

### 3.1 In-scope

- Tournament creation and configuration (name, country list)
- Visual tournament creator with React Flow (drag countries into groups and bracket)
- Group stage view as a React Flow graph (groups → matches → order)
- Knockout bracket view with React Flow, updated as results are entered
- Player management (add, edit, delete)
- Pick registration — always editable, no locks
- Match result registration — always editable by the owner
- Automatic point recalculation on every result change (§ 4 of the rules)
- Standings table with ex aequo support
- Champion and top scorer picks (also always editable)
- Data import / export (JSON)
- Internationalisation (i18next, default language: Polish)
- Auto-update via Tauri Updater + GitHub Releases
- ESLint + Prettier configured for React/TS
- Version bump script (semver)

### 3.2 Out-of-scope

- Pick or result locking
- Login / user roles
- Network sync / backend
- Automatic result fetching from an external API
- Multiple UI languages (i18n ready to extend, but only `pl` for now)

## 6. React Flow — Tournament Visualisation

### 6.1 Three use cases

**A) Tournament creator**  
The user selects a template (e.g. "WC 32 teams") — phases and groups are generated automatically. Then drag & drop countries from the side panel into group nodes on the React Flow canvas. The user can also manually add/remove a phase or rename it before saving.

**B) Group stage view**  
Automatically generated graph after saving the structure. Nodes = groups + matches within a group. Edges connect each group to its matches. Match results can be edited by clicking a match node — always, no locks.

**C) Knockout bracket view**  
Generated automatically based on phase order (`TournamentPhase.order`). Nodes = knockout matches. Edges = winner advances to the next phase. After entering a result the winner propagates to the next node. The user can manually reposition nodes and override edges on the canvas — positions and manual connections are stored in `Match.flowPosition` and `Match.flowEdges` (override the automatic layout).

### 6.2 React Flow node types

```typescript
// Team node (creator — side panel)
type TeamNodeData = {
  countryId: string;
  name: string;
  flag: string;
};

// Group node (creator + group stage view)
type GroupNodeData = {
  groupLabel: string;    // "Group A"
  countryIds: string[];
  phaseId: string;
};

// Match node (group stage + bracket) — phase is dynamic
type MatchNodeData = {
  matchId: string;
  homeTeam: string;      // team name or "TBD"
  awayTeam: string;
  homeGoals?: number;
  awayGoals?: number;
  phaseLabel: string;    // TournamentPhase.label — displayed inside the node
  isKnockout: boolean;   // from TournamentPhase — drives extra-time UI
};

// Phase header node (column header in bracket)
type PhaseHeaderNodeData = {
  phaseId: string;
  label: string;
  order: number;
};
```

### 6.3 Bracketing logic: auto + manual override

**Automatic generation** (based on `TournamentPhase.order`):

```
1. Fetch knockout phases sorted by order ascending
2. For each phase K > first knockout phase:
   - take matches from phase K-1
   - pair winners: match[0].winner → nextMatch.homeTeam, match[1].winner → nextMatch.awayTeam
   - generate React Flow edges between nodes
3. Store node positions (x column = order * columnWidth, y = slot * rowHeight)
```

**Manual override**: The user can drag a node (stores `flowPosition`) or delete/add an edge (stores `flowEdges`). On the next recalculation, automatic connections are ignored for matches that have `flowEdges` set.

**Reset to auto**: A "Reset layout" button clears `flowPosition` and `flowEdges` for all matches in the tournament.

### 6.4 Winner propagation after entering a result

```
saveMatchResult(matchId, result):
  1. Save result on Match
  2. Determine winner = getWinner(result, phase.isKnockout)
  3. Find nextMatch via flowEdges[matchId] or auto-bracket
  4. Set nextMatch.homeTeam or .awayTeam = winner
  5. If nextMatch already had a result → clear it and propagate null further
  6. Recalculate points for all MatchPick records of this match
  7. React Flow re-renders nodes with new data
```

Step 5 is important: editing a quarter-final result can change who plays in the semi-final — so the semi-final result becomes stale and must be cleared (with a warning to the owner).

## 8. Scoring Rules (§ 4 of the Regulations)

### 8.1 Group stage

| Condition | Points |
|---|---|
| Exact score | 3 pts |
| Correct winner or draw (not exact) | 1 pt |
| Wrong result | 0 pts |

```typescript
function calculateGroupPhasePick(pick: MatchPick, result: MatchResult): number {
  if (pick.homeGoals === result.homeGoals && pick.awayGoals === result.awayGoals) return 3;
  const pickSign = Math.sign(pick.homeGoals - pick.awayGoals);
  const resultSign = Math.sign(result.homeGoals - result.awayGoals);
  if (pickSign === resultSign) return 1;
  return 0;
}
```

### 8.2 Knockout stage

- Regular time result (90 min) scored identically to the group stage.
- If the player picked a regular-time draw **and** provided `knockoutWinner`, and it matches the actual winner after extra time / penalties → **+1 bonus point**.
- The extra time / penalty result does not affect regular-time scoring.

```typescript
function calculateKnockoutPhasePick(pick: MatchPick, result: MatchResult): number {
  const base = calculateGroupPhasePick(pick, result);
  const pickIsDrawn = pick.homeGoals === pick.awayGoals;
  const resultIsDrawn = result.homeGoals === result.awayGoals;
  const knockoutWinner = result.penalties?.winner ?? result.extraTime && (
    result.extraTime.homeGoals > result.extraTime.awayGoals ? result.homeTeam : result.awayTeam
  );
  if (pickIsDrawn && resultIsDrawn && pick.knockoutWinner && pick.knockoutWinner === knockoutWinner) {
    return base + 1;
  }
  return base;
}
```

### 8.3 Recalculation on edit

Every change to `MatchResult` (including editing a previously entered result) triggers a full recalculation of points for all `MatchPick` records linked to that match. The standings table always reflects the current state.
